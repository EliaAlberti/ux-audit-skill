# UX Audit — Lumen: Sign-up

> This is an **illustrative example** produced by the `ux-audit` skill against a single
> deliberately-flawed mockup ([`assets/signup.png`](assets/signup.png), generated by
> [`make-example-mockup.py`](make-example-mockup.py)). A real audit would cover the whole journey.

## Executive Summary

The form gets the fundamentals right — two fields only, a single column, and a clear path for returning users — but two majors undercut the one job of this screen, getting a first-time visitor signed up. Field labels exist only as placeholder text, so they vanish the moment the user types (F-01), and the sole piece of error-prevention guidance ("Must be 8+ characters") is rendered at a contrast ratio that is effectively invisible (F-02). A pre-ticked marketing consent box (F-04) is both a dark pattern and a likely GDPR/PECR problem. Top three fixes: add persistent visible labels, fix the helper-text contrast, and untick consent by default.

## Screen-by-Screen

### Step 1 — Sign-up form (`signup.png`)

![annotated](assets/signup-annotated.png)

#### [S3] F-01 · Placeholder-as-label on both fields
- **Check:** FORM-FRICTION · **Heuristics:** WCAG 3.3.2 (Labels or Instructions), Content #1 (front-load), Nielsen #6 (recognition over recall)
- **Evidence:** "Email" and "Password" appear only as grey placeholder text inside the inputs. There is no persistent label above either field.
- **Impact on goal:** Once the user starts typing, the only cue to what a field is for disappears. Mid-form they can no longer tell which box is which without clearing it, and assistive tech often skips placeholder text. Both raise the chance of errors on the one form that gates sign-up.
- **Recommendation:** Add a persistent visible label above each field (or a float-on-focus label). Keep placeholders for format hints only. · **Effort:** S

#### [S3] F-02 · Password requirement is effectively invisible
- **Check:** CONTRAST-FAIL · **Heuristics:** WCAG 1.4.3 (Contrast AA), Nielsen #5 (error prevention)
- **Evidence:** "Must be 8+ characters" sits beneath the password field in a very light grey on white — visibly far below the 4.5:1 AA threshold (verify with a contrast checker).
- **Impact on goal:** This is the only guidance that helps users avoid a password rejection. If they can't read it, they fail validation and bounce off the step that matters most. Error prevention only works if the user can see it.
- **Recommendation:** Raise the helper text to at least 4.5:1 (e.g. #6B7280 on white). Consider showing the rule inline as the user types. · **Effort:** S

#### [S2] F-03 · Primary and secondary actions look identical
- **Check:** CTA-AMBIGUITY · **Heuristics:** Hick's Law, Nielsen #6/#8 (recognition, minimalist emphasis)
- **Evidence:** "Create account" and "Skip for now" are the same size, colour, and weight, side by side. Nothing signals which is the primary path.
- **Impact on goal:** The screen's goal is account creation, yet the design gives equal pull to skipping it. Users pause to choose where they should flow straight through.
- **Recommendation:** Make "Create account" the dominant button (filled, full or wider width); demote "Skip for now" to a text/ghost link. · **Effort:** S

#### [S2] F-04 · Pre-ticked marketing consent
- **Check:** DARK-PATTERN · **Heuristics:** Cialdini (consent obtained by default, not choice), Nielsen #3 (user control)
- **Evidence:** "Send me product news and offers" is ticked by default.
- **Impact on goal:** Beyond eroding trust at the first interaction, pre-ticked consent for marketing is non-compliant under GDPR/PECR, which require a freely-given, affirmative opt-in. Flag for legal as well as UX.
- **Recommendation:** Ship the box unticked. · **Effort:** S

## Journey-Level Findings

- Only the sign-up screen was supplied. The **confirmation / verify-email** screen is not in the set; by the **Peak–End Rule** that ending carries much of the remembered experience, so design it deliberately (clear success state + obvious next step).
- **Fogg B=MAP:** the prompt ("Create account") is present, but **Ability** is taxed by hidden password rules (F-02) and a divided CTA (F-03). Both are cheap to fix and directly lift completion.

## What Works (keep)

- **[P-01] Clear path for existing users** — "Already have an account? Sign in" is present and legible, so returning users aren't trapped on the wrong screen (Jakob's Law, Nielsen #6).
- **[P-02] Minimal field count** — only email and password are requested, keeping Hick's-Law load and form friction low.

## Prioritised Recommendations

1. **Quick wins (sev 3, effort S):** persistent labels (F-01); fix helper-text contrast (F-02)
2. **Quick wins (sev 2, effort S):** untick consent (F-04); make the primary CTA dominant (F-03)
3. **Design the missing ending:** confirmation / verify-email screen (Peak–End)
